## 用户故事

**作为** OneClawShadow 的用户，
**我想要** 管理多个对话会话，每个会话有自己的历史记录和上下文，
**以便** 我可以组织与 AI 的交互，并随时恢复过去的对话。

### 典型场景
1. 用户打开应用，看到空的聊天界面（新对话）。打开导航抽屉查看历史会话。
2. 用户在抽屉中点击"新建对话"开始对话——第一次交换后会话获得自动生成的标题。
3. 用户将会话重命名为更有意义的名称。
4. 用户在抽屉中滑动删除一个会话，然后在 Snackbar 上点击"撤销"恢复它。
5. 用户在抽屉中选择多个会话并批量删除。
6. 用户在离线状态下打开应用，通过抽屉浏览过去的会话和消息历史。

## 功能描述

### 概述
会话管理提供创建、查看、恢复、重命名和删除对话会话的能力。会话是对话的容器——它持有消息历史、当前 Agent 和标题与时间戳等元数据。会话以平铺列表展示，按最后活跃时间排序。会话标题自动生成但可手动编辑。该模块支持单个和批量删除，带有短暂的撤销窗口。

### 会话数据模型

| 字段 | 必填 | 说明 |
|------|------|------|
| ID | 是 | 唯一标识符，自动生成 |
| 标题 | 是 | 会话的显示标题 |

| 当前 Agent ID | 是 | 当前关联到此会话的 Agent |
| 创建时间 | 是 | 会话创建时间戳 |
| 更新时间 | 是 | 最后活动时间戳（最后一条消息发送或接收） |
| 消息数量 | 是 | 会话中的消息数量 |
| 是否活跃 | 是 | 该会话中是否有正在进行的请求 |

### 会话标题生成

会话标题使用两阶段方法生成：

#### 阶段一：即时截取标题
- 用户在新会话中发送第一条消息后，标题设置为该消息的前约 50 个字符，在词边界处截断，必要时附加"..."。
- 这提供了零延迟、零成本的即时标题。

#### 阶段二：AI 生成标题（异步）
- 收到第一条 AI 响应后，发送一个异步后台请求生成更好的标题。
- 请求要求轻量级模型根据第一条用户消息和 AI 响应生成简短标题（5-10 个词）。
- 轻量级模型选择：
  - 已知 provider：使用预设的小模型（例如 OpenAI 用 gpt-4o-mini，Anthropic 用 claude-haiku，Gemini 用 gemini-2.0-flash）
  - 自定义 provider：使用当前对话的模型
- AI 生成的标题就绪后替换截取标题。
- 如果 AI 标题生成失败（网络错误等），保留截取标题。不重试。
- AI 标题生成只发生一次（在第一条 AI 响应后）。之后标题不再自动更新。

#### 手动标题编辑
- 用户可以随时手动编辑会话标题。

### 会话列表

会话展示在**导航抽屉**（从左侧滑入的面板）中，而非独立的首页。通过聊天页面左上角的汉堡菜单图标访问抽屉。详见 [UI 设计规范](../../design/ui-design-spec-zh.md) 的详细布局。

- 会话以平铺列表展示（V1 无文件夹、标签或分组）
- 按最后活跃时间排序（最近的在前）
- 每个列表项显示：
  - 会话标题
  - 当前 Agent 名称（小标签/徽章）
  - 最后消息预览（截断，1-2 行）
  - 最后活跃时间（相对时间："2 分钟前"、"昨天"、"2月20日"）
  - 如果有正在进行的请求，显示活跃指示器
- 抽屉顶部的"新建对话"按钮
- 下拉刷新手势（用于 FEAT-007 同步实现后）

### 会话操作

#### 新建会话
- 导航抽屉顶部的"新建对话"按钮
- 打开使用默认 Agent（通用助手）的空聊天视图
- **延迟会话创建**: 在用户发送第一条消息之前不会在数据库中创建会话记录。这避免了空会话污染会话列表。
- 在第一条消息发送前标题为空/占位符

#### 恢复会话
- 在列表中点击会话打开
- 聊天视图加载完整消息历史
- 用户可以继续对话

#### 重命名会话
- 长按会话或点击编辑图标
- 内联编辑或对话框修改标题


#### 删除单个会话
- 在会话项上滑动删除手势
- 会话立即从列表中移除
- 底部出现 Snackbar："会话已删除"带"撤销"按钮
- 撤销窗口：约 5 秒
- 如果用户点击"撤销"：会话恢复到列表中
- 如果撤销窗口过期：会话及所有消息永久删除

#### 批量删除
- 用户进入选择模式（长按或工具栏操作）
- 每个会话项出现复选框
- 可用"全选"选项
- "删除"按钮删除所有选中会话
- 与单个删除相同的 Snackbar + 撤销机制
- 撤销恢复批次中所有已删除会话

### 离线行为
- 所有会话及其消息历史存储在本地
- 用户可以在离线状态下浏览会话列表和阅读过去的对话
- 用户可以在离线时创建新会话（但在联网前无法发送消息）
- 最后活跃时间和所有元数据在离线时可用

### 用户交互流程

#### 开始新对话
```
1. 用户打开应用（进入空聊天界面，使用通用助手）
   或：用户打开抽屉并点击"新建对话"
2. 用户输入并发送第一条消息
3. 此时在数据库中创建会话（延迟创建）
4. 标题设置为截取的第一条消息（阶段一）
5. AI 响应
6. 后台：发送 AI 标题生成请求（阶段二）
7. 标题就绪时更新为 AI 生成的标题
8. 会话以生成的标题出现在抽屉会话列表中
```

#### 删除会话
```
单个：
1. 用户向左滑动一个会话
2. 会话从列表中消失
3. Snackbar 出现："会话已删除 [撤销]"
4a. 用户点击撤销 -> 会话重新出现
4b. 5 秒过去 -> 会话永久删除

批量：
1. 用户长按一个会话进入选择模式
2. 用户点击更多会话以选中
3. 用户点击工具栏中的"删除"
4. 所有选中会话消失
5. Snackbar："已删除 X 个会话 [撤销]"
6a. 撤销 -> 全部恢复
6b. 超时 -> 全部永久删除
```

## 业务规则

### 会话规则
1. 每个会话必须始终有一个当前 Agent
2. 新会话默认使用通用助手
3. 会话标题永远不为空——至少显示占位符（例如"新对话"）直到发送第一条消息
4. 会话按 `updated_at` 降序排列（最近的在前），V1 中不可由用户配置
5. 删除会话会永久删除其所有消息（撤销窗口过期后）

### 标题生成规则
1. 阶段一在第一条用户消息发送时触发
2. 阶段二在第一条 AI 响应收到后触发
3. 阶段二对已知 provider 使用轻量级模型（gpt-4o-mini、claude-haiku、gemini-2.0-flash）
4. 阶段二对自定义 provider 使用对话的当前模型
5. 阶段二失败时静默保留阶段一标题
6. 标题自动生成只发生一次；之后标题不再自动更新（无论用户是否手动编辑过）
7. 标题生成不阻塞对话流程

### 删除规则
1. 删除的会话在撤销窗口期间（约 5 秒）为软删除
2. 撤销窗口后：硬删除（消息和会话记录从数据库中移除）
3. 撤销恢复会话到精确的之前状态
4. 批量撤销是全有或全无（恢复全部或不恢复）
5. 如果应用在撤销窗口期间被杀死，软删除的会话在下次应用启动时被清理

## 非功能性需求

### 性能
- 会话列表加载时间 < 200ms（最多 1000 个会话）
- 会话创建即时完成（< 50ms）
- AI 标题生成在 5 秒内完成（后台，非阻塞）
- 滚动会话列表保持 60fps

### 数据
- 所有会话数据存储在本地（SQLite/Room）
- 会话数据包含在 Google Drive 同步中（FEAT-007）
- 消息历史在打开会话时懒加载（不为所有会话预加载）

### 可靠性
- 如果 AI 标题生成请求正在进行而应用被杀死，截取标题保留
- 意外应用终止不会丢失数据（消息到达时即持久化）

## 错误处理

### 错误场景

1. **AI 标题生成失败**
   - 处理：静默保留截取标题，不向用户显示错误
   - 不重试（避免不必要的 API 费用）

2. **会话创建失败（存储错误）**
   - 展示：错误 toast"创建会话失败。请重试。"
   - 不打开聊天视图

3. **会话删除失败（存储错误）**
   - 展示：错误 toast"删除会话失败。"
   - 会话保留在列表中

4. **会话恢复失败（数据损坏）**
   - 展示：错误对话框"无法加载此会话。"
   - 选项：删除损坏的会话

5. **离线标题生成**
   - 阶段一（截取）在离线时正常工作
   - 阶段二（AI 生成）离线时跳过——回到在线后不重试
