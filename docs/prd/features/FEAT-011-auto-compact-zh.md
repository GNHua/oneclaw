## 用户故事

**作为** OneClaw 的用户,
**我想要** 应用自动管理对话上下文长度，使长对话不会因为超出模型上下文窗口而失败,
**以便** 我可以进行长时间对话，无需担心技术限制或丢失重要上下文。

### 典型场景
1. 用户进行一次长对话（50+ 条消息）。累积的 token 数接近模型上下文窗口限制。应用在下一次请求前自动将较早的消息压缩为摘要，对话无缝继续。
2. 用户让 AI 使用 HTTP 请求工具获取一个大型网页。工具返回结果（几百 KB）在存储和发送给模型前被自动截断到合理大小。
3. 用户在会话中途从 200K 上下文窗口的模型切换到 8K 的模型。应用检测到下一条消息会超出新模型的限制，在发送前触发 compact。

## 功能描述

### 功能概述
Auto Compact 是一个上下文窗口管理系统，防止对话超出模型的上下文窗口限制。当累积的输入 token 接近可配置的阈值（默认：模型上下文窗口的 85%）时，应用使用当前模型自动总结较早的消息，并在后续 API 请求中用摘要替换它们。原始消息在数据库中保持不变。

此外，过大的工具调用结果（如获取网页）会在存储时被截断，防止数据库膨胀，并确保工具结果不会占用过多的上下文空间。

### 详细说明

#### Auto Compact

**触发条件**：每次 API 响应完成后（包括所有工具调用轮次），系统检查对话历史的总 token 数是否超过当前模型上下文窗口的 85%。

**处理流程**：
1. 计算会话中所有消息的总 token 数
2. 如果总数超过阈值，确定需要压缩的消息：
   - 从最新的消息开始向前遍历，累加 token 数
   - 累积 token 数在上下文窗口 25% 以内的消息被"保护"（保持原样）
   - 所有更早的消息作为压缩候选
3. 向当前模型发送总结请求，包含候选消息（加上任何已有的先前摘要）
4. 将生成的摘要存储在 Session 的 `compactedSummary` 字段中
5. 记录哪些消息已被压缩（通过存储压缩边界的时间戳或消息 ID）

**后续请求**：构建 API 请求的 `messages` 数组时，系统将 `compactedSummary` 作为系统级上下文消息前置，然后附上仅未压缩的（最近的）消息。

**多次压缩**：如果一个会话经历多次压缩，每次新的压缩会将先前的 `compactedSummary` 加上新压缩的消息合并为一条新的摘要。Session 始终只保存一条累积摘要。

#### Tool Result 截断

**触发条件**：在将工具结果消息存入数据库之前。

**处理流程**：
1. 检查工具结果内容的字符长度
2. 如果超过截断限制（默认：30,000 字符），从尾部截断
3. 追加截断标记：`\n\n[... content truncated, showing first {kept} characters of {total} total ...]`
4. 将截断后的版本存入数据库

截断只在存储时发生一次。所有下游消费者（API 请求、UI 显示）使用已截断的版本。

### 用户交互流程

#### Auto Compact（基本不可见）
```
1. 用户发送消息并收到响应
2. 系统检查 token 数是否超过阈值
3. 如果超过阈值：
   a. 系统在后台发送总结请求
   b. 显示简短提示（如"正在优化对话上下文..."）
   c. 摘要存储到 Session 中
   d. 提示消失
4. 用户正常继续聊天
5. 如果 compact 重试后仍失败，系统回退到截断
   并显示 Snackbar："对话历史已自动精简"
```

#### Tool Result 截断（不可见）
```
1. AI 调用工具（如 HTTP 请求获取网页）
2. 工具执行并返回大型结果
3. 系统在存储前截断结果
4. 截断后的结果显示在工具调用消息中
5. 截断后的结果发送回模型
```

## 验收标准

必须满足的标准（所有必须通过）：
- [ ] AiModel 实体包含 `contextWindowSize` 字段（Int，可空）
- [ ] 预置模型已填充正确的 `contextWindowSize` 值
- [ ] 当会话总 token 数超过模型上下文窗口的 85% 时，自动触发 compact
- [ ] Compact 通过调用当前模型生成摘要
- [ ] 摘要存储在 `Session.compactedSummary` 字段中
- [ ] 原始消息永远不会被 compact 修改或删除
- [ ] 后续 API 请求包含 compact 摘要 + 仅最近的（未压缩的）消息
- [ ] 同一会话中多次压缩合并为一条累积摘要
- [ ] Compact 期间显示简短的 UI 提示
- [ ] 如果总结 API 调用失败，静默重试一次
- [ ] 如果重试也失败，系统回退到截断（丢弃最早的消息）并显示 Snackbar
- [ ] 超过 30,000 字符的工具结果在存储前被截断
- [ ] 截断的工具结果包含截断标记
- [ ] 85% 阈值定义为常量（不内联硬编码）
- [ ] 30,000 字符截断限制定义为常量

可选标准（Nice to have）：
- [ ] 用户可以从 UI 手动触发 compact
- [ ] 聊天界面显示 token 使用量 vs 上下文窗口
- [ ] 可在设置中配置阈值百分比

## UI/UX 要求

### Auto Compact 提示
- Compact 期间：微妙的非阻塞提示（如最新 AI 响应下方的小进度消息或短暂的 Snackbar 风格横幅）
- 文本："正在优化对话上下文..."
- 持续时间：仅在总结 API 调用期间可见
- 提示不得阻塞用户输入 -- 用户可以在 compact 运行时继续打字

### Compact 回退通知
- 当 compact 失败并回退到截断时：Snackbar 通知
- 文本："对话历史已自动精简"
- 持续时间：标准 Snackbar 持续时间（短）

### Tool Result 截断
- 无需 UI 提示 -- 截断对用户不可见
- 如果用户展开工具调用详情视图，截断标记文本作为工具结果的一部分显示

## 功能边界

### 包含的功能
- 基于 token 阈值触发的自动上下文压缩
- 基于 token 数的保护窗口（最近 25% 的上下文）
- 使用当前对话模型生成摘要
- 在 Session 实体上存储累积摘要
- 摘要失败时回退到消息截断
- 存储时的工具结果截断
- AiModel 实体上的上下文窗口大小字段

### 不包含的功能
- 用户可配置的 compact 阈值设置（未来）
- 手动 compact 触发按钮（未来）
- 客户端 token 计数 / tokenizer（依赖 API 报告的 token 数）
- 单条消息内部压缩（仅整条消息粒度）
- 工具结果流式传输或分页
- 从 API 自动检测上下文窗口大小（使用预配置值）

## 业务规则

### Compact 规则
1. Compact 仅在完整 API 响应后触发（包括所有工具调用轮次），从不在流式传输中触发
2. 85% 阈值基于当前模型的 `contextWindowSize` 计算
3. 如果 `contextWindowSize` 为 null（未知），compact 不触发（无操作）
4. 保护窗口（最近消息）按 token 数目标为上下文窗口的 25%
5. 总结请求使用与当前对话相同的模型、提供商和 API 密钥
6. 摘要 prompt 是一个固定的系统提示，指示模型生成简洁的事实摘要
7. Compact 是幂等的 -- 在已低于阈值时触发是无操作

### Token 计数规则
1. Token 数来自 API 报告的 `Usage` 事件，存储在每条消息上（`tokenCountInput`、`tokenCountOutput`）
2. 对于没有 token 数的消息（如尚未发送的用户消息），使用基于字符的估算（每 4 个字符 1 个 token）
3. 阈值比较的总 token 数是下一次 API 请求中所有将要发送的消息的估算输入 token 之和

### Tool Result 截断规则
1. 截断限制：每个工具结果 30,000 字符
2. 截断保留内容开头（head），丢弃尾部
3. 追加截断标记以表明内容已被截断
4. 截断在数据库插入前发生 -- 完整内容永远不会被存储
5. 截断平等地应用于所有工具类型

### 失败处理规则
1. 如果总结失败，精确重试一次（相同请求）
2. 如果重试失败，回退到截断：从 API 请求中移除最早的消息（但不从数据库中删除），直到低于 85% 阈值
3. 在回退时显示 Snackbar 通知
4. 记录失败日志以供调试

## 非功能性需求

### 性能要求
- Compact 应在 15 秒内完成（总结 API 调用）
- Compact 不得阻塞 UI 线程或阻止用户输入
- 工具结果截断是同步字符串操作，必须在 < 10ms 内完成
- 每次响应后的 token 数检查应增加 < 5ms 开销

### 可靠性
- Compact 失败绝不能阻止用户继续对话
- 工具结果截断绝不能抛出异常
- 系统必须处理边缘情况：空会话、仅有用户消息的会话、所有消息都在保护窗口内的会话

### 数据完整性
- 原始消息永远不会被 compact 修改或删除
- `compactedSummary` 字段可以被清除，如果用户想"重置"上下文（未来功能）
- 数据库大小通过工具结果截断来限制

## 错误处理

### 可能的错误场景

1. **总结 API 调用失败（网络错误）**
   - 操作：静默重试一次
   - 如果重试失败：回退到截断，显示 Snackbar
   - 对话正常继续

2. **总结 API 返回空或格式错误的响应**
   - 操作：视为失败，回退到截断
   - 显示 Snackbar 通知

3. **模型上下文窗口大小未知（null）**
   - 操作：完全跳过 compact（无操作）
   - 无需用户通知

4. **所有消息都在保护窗口内**
   - 操作：无需 compact（无操作）
   - 当对话消息少但最近的消息很长时可能出现

5. **工具结果截断边缘情况（内容恰好在限制值）**
   - 操作：无需截断，原样存储

## 测试要点

### 功能测试
- 验证 token 数超过 85% 阈值时触发 compact
- 验证低于阈值时不触发 compact
- 验证模型上下文窗口为 null 时不触发 compact
- 验证保护窗口计算（按 token 数的最近 25%）
- 验证摘要存储在 Session.compactedSummary 中
- 验证原始消息在 DB 中保持不变
- 验证后续 API 请求使用摘要 + 仅最近消息
- 验证多次压缩合并为一条累积摘要
- 验证 compact 失败重试（一次重试）
- 验证双重失败时回退到截断
- 验证回退时显示 Snackbar
- 验证 30K 字符时的工具结果截断
- 验证追加截断标记
- 验证低于 30K 的工具结果不被修改

### 边缘情况
- 0 条消息的会话
- 仅 1 条用户消息的会话
- 所有消息都在保护窗口内的会话
- 恰好 30,000 字符的工具结果
- 包含多字节 Unicode 字符的工具结果
- 会话中途切换到更小上下文窗口的模型
- Compact 已在进行中时再次触发（不得重复 compact）
- 已有先前 compact 的 compactedSummary 的会话

### 性能测试
- 100+ 条消息时的 compact 完成时间
- 每次消息发送时的 token 计数开销
- 1MB+ 内容时的工具结果截断速度
