## 用户故事

**作为** OneClawShadow 的用户，
**我想要** AI Agent 能够跨会话记住重要的上下文、偏好和历史决策，
**以便** 我不需要重复自己，Agent 随着使用变得越来越有帮助。

### 典型场景
1. 用户在一个会话中告诉 Agent"我喜欢简洁的回答"。在后续会话中，Agent 记住了这个偏好，无需提醒就保持简洁回复。
2. 用户就一个项目进行了长时间的对话。第二天，Agent 能够回忆起做出的关键决策并从上次中断的地方继续。
3. 用户问"上周我们讨论的 API 设计是什么？"，Agent 检索相关的每日日志条目来回答。
4. 用户手动编辑长期记忆文件，修正或添加 Agent 应该始终知道的信息。

## 功能描述

### 概述
Agent 记忆系统为 OneClawShadow 提供持久化的跨会话记忆能力。目前，应用将完整的消息历史存储在 Room 数据库中，每次会话都发送全部历史 -- 没有任何机制让 Agent 回忆前次会话的信息，也没有超出原始消息历史的上下文管理。

本功能引入两层记忆（每日日志和长期记忆），以 Markdown 文件的形式存储在设备上，并配备混合搜索引擎（BM25 关键词 + 向量语义）来检索相关记忆并注入到系统提示中。一个小型本地嵌入模型在设备上运行进行向量搜索，无需 API 调用。

### 详细说明

#### 每日日志
- 在每个会话结束时（以及其他触发时机），AI 将对话要点摘要到按日期组织的 Markdown 文件中。
- 每个日志文件以日期命名（例如 `2026-02-28.md`），存储在 `getFilesDir()/memory/daily/` 目录下。
- 摘要由当前活跃的 AI 模型使用专用摘要提示词完成。
- 每个会话维护一个 `lastLoggedMessageId` 指针，防止对已处理的消息重复摘要。
- 每日日志捕获：讨论的关键主题、做出的决策、完成的任务、用户表达的偏好以及值得注意的事实。

#### 长期记忆（MEMORY.md）
- 一个持久化的单一文件（`getFilesDir()/memory/MEMORY.md`），包含稳定的事实、用户偏好和 Agent 应始终可访问的关键知识。
- 通过两种方式填充：
  - **AI 自动提取**：在写入每日日志时，AI 识别应提升为长期记忆的事实/偏好并追加到文件中。
  - **用户手动编辑**：用户可以通过设置界面直接查看和编辑 MEMORY.md。
- MEMORY.md 始终包含在系统提示中（如果文件较大，则包含前 N 行），为每次对话提供基础上下文。
- 按主题使用 Markdown 标题进行语义组织。

#### 混合搜索
- 当用户发送消息时，记忆系统搜索相关的历史上下文。
- **BM25 关键词搜索（30% 权重）**：对每日日志条目和记忆块进行全文搜索，擅长精确匹配和特定术语。
- **向量语义搜索（70% 权重）**：使用余弦距离的基于嵌入的相似度搜索，擅长基于含义的检索。
- **时间衰减**：越近的记忆权重越高。分数乘以基于时间的衰减因子。
- 结果合并、排序，选择 Top-K 最相关的记忆块进行注入。

#### 本地嵌入模型
- 一个小型 Transformer 模型（约 22MB，例如 MiniLM-L6-v2）通过 ONNX Runtime 在设备上运行。
- 生成 384 维嵌入向量，用于记忆块和查询。
- 无需 API 调用 -- 所有嵌入计算都在本地完成。
- 模型资源随应用打包（或首次启动时下载）。

#### 记忆注入
- 检索到的记忆被格式化后注入到系统提示中，位于 Agent 自定义系统提示之下。
- 格式：一个 `## Relevant Memories` 区块，包含 Top-K 结果，每条附带来源标注（每日日志日期或 MEMORY.md）。
- 无论搜索结果如何，MEMORY.md 内容（或前 200 行）始终被注入。

#### 触发机制
每日日志写入由以下事件触发：
1. **会话结束**：用户主动结束或切换离开一个会话。
2. **应用进入后台**：应用进入后台时（通过 `ProcessLifecycleOwner`）。
3. **会话切换**：用户从一个会话切换到另一个会话。
4. **日期变更**：活跃会话期间日期发生变化。
5. **压缩前刷写**：在 FEAT-011 自动压缩消息历史之前，先提取重要上下文保存到每日日志。

#### 消息追踪
- 每个会话跟踪一个 `lastLoggedMessageId`，表示已处理的用于每日日志提取的最后一条消息。
- 当每日日志触发器触发时，只有 `lastLoggedMessageId` 之后的消息会被发送到摘要模型。
- 这防止了重复条目并确保高效的增量日志记录。

### 用户交互流程
```
每日日志（自动）：
1. 用户在会话中进行对话
2. 会话结束 / 应用进入后台 / 日期变更
3. 系统提取 lastLoggedMessageId 之后的新消息
4. 系统调用 AI 模型对对话片段进行摘要
5. 摘要追加到每日日志文件（例如 2026-02-28.md）
6. AI 识别长期事实/偏好并追加到 MEMORY.md
7. 更新 lastLoggedMessageId

记忆检索（自动）：
1. 用户在任意会话中发送新消息
2. 系统使用本地模型对用户消息生成嵌入
3. 系统运行混合搜索（BM25 + 向量）查询记忆索引
4. 检索 Top-K 最相关的记忆
5. 记忆 + MEMORY.md 内容注入系统提示
6. 消息连同丰富的上下文一起发送到 AI 模型

手动记忆编辑：
1. 用户导航到 设置 > 记忆
2. 用户在编辑器中查看 MEMORY.md 内容
3. 用户添加、编辑或删除条目
4. 保存更改并更新搜索索引
```

## 业务规则

### 记忆规则
1. 每日日志在一天之内是只追加的 -- 新摘要追加写入，旧条目不会被自动修改。
2. MEMORY.md 可以被 AI（追加）和用户（完整编辑）修改。
3. 如果触发器触发时 AI 模型不可用，每日日志写入静默跳过（不崩溃，不显示错误）。
4. 记忆注入不得超过可配置的 token 预算（默认：2000 tokens），以避免占用过多上下文窗口。
5. 本地嵌入模型在首次使用时惰性加载，而非应用启动时加载。

### 数据规则
- 每日日志文件：每天一个文件，Markdown 格式，命名为 `YYYY-MM-DD.md`
- MEMORY.md：单一文件，Markdown 格式，按主题标题组织
- 搜索索引：Room 数据库表，映射块 ID 到嵌入和元数据
- 嵌入向量：384 维浮点向量，以 BLOB 形式存储在 Room 中

### 存储规则
- 记忆文件存储在 `getFilesDir()/memory/`（内部应用存储，应用更新后保留）
- 每日日志：`getFilesDir()/memory/daily/`
- 长期记忆：`getFilesDir()/memory/MEMORY.md`
- 嵌入模型资源：`assets/models/` 或下载到 `getFilesDir()/models/`

## 非功能性需求

### 性能要求
- 单次查询嵌入：中端设备 < 200ms
- 混合搜索（查询 + 排序）：总计 < 500ms
- 每日日志摘要：后台运行，无 UI 影响
- 记忆注入为消息发送流程增加 < 100ms
- 本地嵌入模型推理时内存占用：< 100MB

### 存储要求
- 嵌入模型大小：约 22MB（ONNX 格式）
- 每日日志文件：每天约 1-5KB
- 搜索索引：随内容增长，估计日常使用一年 < 10MB
- MEMORY.md：典型使用预期 < 50KB

### 可靠性
- 如果嵌入模型加载失败，回退到仅 BM25 搜索
- 如果每日日志写入失败，记录错误并继续 -- 不崩溃应用
- 如果记忆注入失败，不带记忆上下文发送消息而非阻塞

## 错误处理

### 错误场景

1. **嵌入模型加载失败**
   - 回退：使用仅 BM25 搜索（无向量组件）
   - 记录警告，不向用户显示错误
   - 下次应用启动时重试加载

2. **每日日志触发时 AI 模型不可用**
   - 跳过本次触发的每日日志写入
   - 未处理的消息将在下次触发时被提取（lastLoggedMessageId 不更新）
   - 不向用户显示错误

3. **记忆文件写入失败（磁盘已满、权限错误）**
   - 记录错误
   - 如果持续发生则显示低调的通知（例如磁盘空间不足警告）
   - 不崩溃

4. **搜索索引损坏**
   - 在设置中提供"重建索引"按钮
   - 检测到损坏时，回退到仅文件搜索直到重建完成

5. **MEMORY.md 变得很大（> 200 行）**
   - 仅前 200 行注入到系统提示中
   - 建议用户通过设置编辑器整理/精简

| 数据项 | 类型 | 必填 | 说明 |
